De opdracht

Gebruik een externe API om een zinvolle, interactieve web-applicatie te bouwen. Het doel: laten zien dat je data kunt ophalen, verwerken, weergeven en dat je de API goed genoeg begrijpt om er creatief mee om te gaan.

Het concept: F1 Explorer

F1 Explorer haalt live F1-sessiedata op via de OpenF1 API en visualiseert dit als een interactief tijdlijn-replay. Je kiest een race, kwalificatie of vrije training uit de seizoenskalender, klikt op "Load Track" en kijkt de sessie terug real-time of tot 50× versneld. Elke rijder is een stip op het circuit, met echte GPS-coördinaten als basis.

OpenF1 API Three.js (WebGL) Astro Rate-limited fetcher Orthographic camera Canvas telemetry HUD sessionStorage BufferGeometry trails Web Audio API TextureLoader headshots

Wat de site kan

Live track replay

GPS-punten van alle rijders geplot op een Three.js orthografisch circuit. Stippen met rijderskleur, foto als texture en een fading trail van de afgelopen 4 seconden. Verstelbare snelheid: 1× tot 50×.

Race calendar

Seizoenskalender voor 2023, 2024 en 2025 opgehaald via de OpenF1 API. Elk Grand Prix-weekend heeft klikbare sessie-pills (FP1, FP2, FP3, Q, SPR, R).

Live towers

Live intervals (race), kwalificatiescore per fase (Q1/Q2/Q3), praktijktijden, live lap tracker, pit-stop tower, inhaalacties en race control berichten allemaal bijgehouden tot op de gesimuleerde tijdlijn.

Team radio

Teamradio-berichten spelen automatisch af op het juiste moment in de tijdlijn. Popup met rijderskleur, play/pause en voortgangsbalk.

Telemetry HUD

Klik een rijder aan voor een canvas-HUD: snelheid, RPM, throttle, rem, versnelling en DRS live bijgehouden via de car_data endpoint. Radiale arc-bars voor RPM en throttle.

Podium screen

Aan het einde van een race verschijnt automatisch een podium-overlay met P1/P2/P3 headshots en de snelste ronde.

DRS zones

DRS-activatie uit de car_data wordt gemapped naar het circuit en gevisualiseerd als lichtgroene segmenten met glow-effect.

F1-thema bij laden

Tijdens het laden speelt het officiële F1-thema voor het sessiejaar. Fade-in bij start, fade-out zodra de sessie geladen is.

Code highlight: rate-limited API fetcher

De OpenF1 API heeft strikte limieten: max 3 requests per seconde en 28 per minuut. In plaats van gewoon fetch() aan te roepen heb ik een wrapper gebouwd die een rolling request-log bijhoudt en automatisch wacht als een limiet bereikt wordt: 

// track-data.js automatisch wachten bij rate limits
export function createApiFetcher(onRateLimitMessage) {
  const requestLog = [];
  const PER_SEC = 3, PER_MIN = 28;
  const delay = ms => new Promise(r => setTimeout(r, ms));

  return async function apiFetch(url) {
    while (true) {
      const t = Date.now();
      while (requestLog.length && requestLog[0] < t - 60_000)
        requestLog.shift();

      const recentSec = requestLog.filter(ts => ts > t - 1_000).length;
      if (recentSec >= PER_SEC) {
        const oldest1s = requestLog.find(ts => ts > t - 1_000);
        await delay(1_000 - (t - oldest1s) + 50);
        continue;
      }
      if (requestLog.length >= PER_MIN) {
        await delay(60_000 - (t - requestLog[0]) + 50);
        continue;
      }
      requestLog.push(Date.now());
      const res = await fetch(url);
      if (res.status === 429) {
        requestLog.pop();
        await delay(10_000);
        continue;
      }
      return res.json();
    }
  };
}

Code highlight: driver trails via BufferGeometry

Elke rijder heeft een fading trail van de afgelopen 4 seconden. In plaats van steeds nieuwe geometrie aan te maken, hergebruik ik een vaste BufferGeometry met 41 vertices. Kleur per vertex fadet van transparant (staart) naar de teamkleur (kop): 

// track-overlays.js vertex colors: transparant (staart) → teamkleur (kop)
const TRAIL_POINTS = 40;

const geo = new THREE.BufferGeometry();
geo.setAttribute('position', new THREE.BufferAttribute(positions, 3));
geo.setAttribute('color',    new THREE.BufferAttribute(colors,    3));

// Elke frame updaten geen nieuwe objecten
for (let i = 0; i <= TRAIL_POINTS; i++) {
  const alpha = i / TRAIL_POINTS; // 0=onzichtbaar → 1=vol
  colors[i * 3]     = rgb.r * alpha;
  colors[i * 3 + 1] = rgb.g * alpha;
  colors[i * 3 + 2] = rgb.b * alpha;
}
geo.attributes.color.needsUpdate = true;

Code highlight: Doctor Who easter egg

Typ who op het toetsenbord en alle rijders op de grid worden vervangen door Doctor Who-karakters. Portretten worden live opgehaald via de Wikipedia pageimages API. Een tweede who zet alles terug: 

// easter-egg-who.js Wikipedia thumbnails ophalen en rijderinfo vervangen
const WHO_CHARACTERS = [
  { name: 'The Doctor', abbr: 'DOC', colour: '#003B6F', wiki: 'Tenth_Doctor' },
  { name: 'The Dalek',  abbr: 'DAL', colour: '#B8860B', wiki: null,
    img: 'https://upload.wikimedia.org/...' },
  // ... 18 meer karakters
];

async function activate() {
  const pool   = shuffle(WHO_CHARACTERS).slice(0, driverNums.length);
  const thumbs = await fetchWikiThumbnails(pool.filter(c => c.wiki).map(c => c.wiki));

  driverNums.forEach((dn, i) => {
    const char = pool[i];
    Object.assign(driverInfoMap[dn], {
      acronym: char.abbr, lastName: char.name,
      colour: char.colour,
      headshotUrl: char.img ?? thumbs[char.wiki] ?? null,
    });
  });
  // Three.js dots live herladen met nieuwe texturen
  driverDots.forEach((dot, idx) => reloadDotTexture(dot.dotMesh, ...));
}

Nieuwe inzichten

01

API rate limiting is een eerste-klas probleem

Een API aanroepen is makkelijk. Een API aanroepen zonder ban te krijgen terwijl je 15+ endpoints parallel bevraagt, is een architectuurprobleem. De rolling-window rate limiter was meer werk dan verwacht en meer waard dan ik dacht.

02

Three.js voor data visualisatie

WebGL via Three.js is niet alleen voor 3D games. Een orthografisch cameramodel, BufferGeometry voor trails en TextureLoader voor rijderfoto's Three.js bleek een uitstekende keuze voor een 2D data-visualisatie die soepel moet lopen.

03

Module-splitting als architectuurdiscipline

22 JS-modules, elk met één verantwoordelijkheid. De verleiding om alles in één file te stoppen was groot. De splitsing hield het project behapbaar en maakte het makkelijk om losse towers te testen en uitbreiden.

04

Tijdlijn-gebaseerde state is complex

Elke tower, elke trail, elke sector ze moeten allemaal consistent zijn met de gesimuleerde tijd. Scrubben naar een eerder moment, dan terug spelen. Stationaire rijders verbergen. Dat soort randgevallen doe je pas echt goed als je ze allemaal een keer fout hebt.

05

Easter eggs verdienen architectuur

De Doctor Who easter egg vervangt rijdersinfo live in Three.js inclusief het herladen van texturen in bestaande dot-meshes. Dat vereiste een clean interface tussen de easter egg module en de rest van het systeem.

06

Astro als lichte app-shell

Astro laat je HTML componenten schrijven met zero JS-overhead. De zware interactiviteit zit in vanilla JS modules Astro doet alleen de structuur. Die scheiding hield de site snel en de code overzichtelijk.

Reflectie

Dit was het project waarbij ik voor het eerst echt in een externe API ben gedoken op een schaal die er toe doet. Niet één endpoint, maar dertien. Niet één rijder, maar twintig tegelijk, met echte GPS-data die tot 50× versneld op het scherm rijdt.

Het moment waarop de eerste stippen over het circuit bewogen was een van de beste momenten van de minor. Daarna werd het pas echt interessant: rate limiting, stationary detection, de stint-gebaseerde flying lap filter, de canvas telemetry HUD. Elk probleem was kleiner dan het volgende.

De Doctor Who easter egg was bewust overspecs. Niet gevraagd, wel leuk om te bouwen. En het paste goed bij de opdracht: laat zien dat je de API begrijpt genoeg om er creatief mee om te gaan.

Learning goals

  • Learn how to effectively use CSS: Voor data-gedreven pagina's heb ik overzichtelijke tabellen en responsive kaarten gebouwd. Ik ga CSS-utilities voor datastructuren toevoegen aan mijn toolkit om herbruikbaarheid te vergroten. Dat maakt het makkelijker om nieuwe datasets te presenteren en te onderhouden.
  • Learn how to animate using CSS: Animaties voor laadstaten en skeletons verbeterden de perceptie van performance. Ik ga deze patronen standaardiseren in data-fetch workflows. Hierdoor lijkt content sneller en gebruikers blijven betrokken tijdens laadmomenten.
  • Be more creative in designing: Visualisatie van API-data wordt sterker met kleine grafische elementen. Ik test eenvoudige chart-embeds en kleurencoderingen om patronen zichtbaar te maken. Dit helpt bij snelle interpretatie van cijfers en maakt insights directer bruikbaar.
