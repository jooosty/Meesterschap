Killian Valkhof

CSS as a replacement for JavaScript

Rule of Least Power CSS-only interactie Backward compatibility Input styling Accordeons & dialogs

Wat hij vertelde

Killian opende met een principe dat me de hele minor is bijgebleven: de Rule of Least Power — kies altijd de minst krachtige taal die geschikt is voor een taak. Voor het web betekent dat: kijk eerst naar HTML, dan CSS, dan pas JavaScript.

"Je kan meer CSS gebruiken voor dingen dan je denkt, in plaats van JS."

Browsers zijn backwards compatible — als iets ooit heeft gewerkt, moet het altijd blijven werken. Dat is een belofte die browsers aan het web doen. Killian liet zien dat je dat als voordeel kunt gebruiken: CSS die je leert, hoef je nooit opnieuw te leren.

appearance: none

Haalt alle browser-styling weg van een input, zodat je hem volledig zelf kunt stylen. Inputs waren 10 jaar geleden moeilijk te stylen — nu niet meer.

scroll-behaviour: smooth

Smooth scrollen in puur CSS, zonder JavaScript. De browser bepaalt de animatie en respecteert automatisch prefers-reduced-motion.

scroll-margin

Als je een fixed header hebt en smooth scrollt naar een anker, zorgt scroll-margin dat de pagina stopt met de juiste afstand tot de top.

<details> & accordeons

Native accordeon zonder JavaScript. Het name attribuut zorgt dat maar één tegelijk open kan staan. ::marker overschrijft het pijltje. <details open> forceert de open staat.

<dialog>

De moderne vervanger van alert(). Je kunt hem stylen — alert niet. Standaard onzichtbaar, needs JS om te openen — maar de rendering en focus-trap zijn gratis.

prefers-reduced-motion

Gebruikers kunnen in het OS aangeven dat ze minder beweging willen. @media (prefers-reduced-motion: reduce) respecteert dit in CSS.

Een scherpe observatie: als je iets leert, check dan niet alleen of het werkt — maar of er iets beter bestaat. Kennis is geen eindbestemming.

Wat ik heb meegenomen

De Rule of Least Power herkende ik direct in KRAFTWERK (CSS to the Rescue). Maar het echte inzicht was breder: elke keer als ik nu JavaScript schrijf, vraag ik me eerst af of HTML of CSS het ook kan. Vaak wel.

19 februari 2026

Peter-Paul Koch

Browsers & support detection

Browser architectuur Rendering engine Render blocking Backward compatibility Browsergeschiedenis

Wat hij vertelde

Peter-Paul Koch — auteur van quirksmode.org en een van de meest ervaren browser-experts ter wereld — gaf een lezing over hoe browsers eigenlijk werken en hoe we daarmee moeten omgaan als developers.

Wat zit er in een browser?

Rendering engine

Verantwoordelijk voor HTML en CSS parsing, rendering, en de constructie van de DOM en AOM-trees. Niet voor de interface of JavaScript engine. Chrome gebruikt Blink (afgesplitst van WebKit).

AOM — Accessibility Object Model

Naast de DOM bouwt de browser ook een AOM — een parallelle boom die screenreaders en andere assistive technology gebruiken om de pagina te begrijpen.

Core vs DOM

Core: wat JavaScript een programmeertaal maakt (variabelen, functies, loops). DOM: de manier waarop JS delen van een HTML-pagina kan aanspreken en wijzigen. Twee aparte dingen.

Render blocking

CSS blokkeert altijd rendering. JavaScript ook — tenzij je async of defer gebruikt. Defer: laad later, voer uit als DOM klaar is. Async: laad en voer uit zodra klaar, ongeacht DOM.

Een korte browsergeschiedenis

PPK nam ons mee door de browseroorlog — en hoe die het web heeft gevormd:

1990: WorldWideWeb — de allereerste browser, gebouwd door Tim Berners-Lee. 1993: Mosaic — eerste browser voor gewone mensen. 1994: Netscape — eerste betaalde browser. 1995: Internet Explorer & Opera (Opera's oprichter was ook uitvinder van CSS). 2000: Konqueror — hobbyproject van Linux-gebruikers, engine later opgekocht door Apple. 2003: Safari. 2004: Firefox. 2007: Safari iOS. 2008: Chrome met de Blink-engine. 2024: Ladybird — nieuwe rendering engine, gestart als hobbyproject.

"Browsers are the most hostile development platforms in the world."

— Douglas Crockford (maker van JSON)

Backward compatibility: als iets ooit heeft gewerkt in een browser, moet het altijd blijven werken. Geen doctype in je HTML? Dan valt de browser terug op Internet Explorer 6-gedrag.

Wat ik heb meegenomen

Ik kende het verschil tussen DOM en Core niet echt — ik gebruikte beide gewoon als "JavaScript". Het AOM-model was nieuw voor mij en sloot direct aan op het HCD-vak: de browser bouwt een aparte boom voor screenreaders. Als je die negeert in je HTML, negeert de screenreader jouw content.

Nils Binder

9Elements · Design voor het web

Figma vs Photoshop Web development triangle Design tools CSS Grid live

Wat hij vertelde

Nils Binder werkt bij 9Elements — een bureau dat communications design, product development en web development combineert. Ze werkten onder andere voor Schalke 04 en de OECD.

Hij nam ons mee door hoe het designproces voor het web is veranderd: van Photoshop (20 jaar geleden) naar Figma (nu). Photoshop was gemaakt voor printontwerp en deed dingen die op het web helemaal niet mogelijk waren. Figma is gebouwd voor het web en sluit veel beter aan op wat browsers kunnen.

Figma vs Photoshop

Photoshop: overcompliceerd, gemaakt voor print, makkelijk dingen ontwerpen die het web niet ondersteunt. Figma: eenvoudiger, variabelen, dichter bij hoe het web werkt.

Web development triangle

De huidige workflow bij veel bureaus: Figma → Tailwind → AI. Nils liet zien hoe dat de drempel verlaagt maar ook risico's meebrengt voor kwaliteit en begrip.

CSS Grid live

Nils demonstreerde live hoe je een nieuwspagina-layout bouwt met grid-template-columns: 2fr 3fr auto 1fr — hoe weinig code je nodig hebt als je CSS Grid echt begrijpt.

"In Figma is alles mogelijk wat op het web kan. Maar niet alles wat op het web kan, is beschikbaar in Figma."

Wat ik heb meegenomen

De web development triangle (Figma → Tailwind → AI) zette me aan het denken. Het maakt dingen sneller, maar het vergroot ook de kans dat je nooit echt begrijpt waarom iets werkt. De minor is een antwoord op die driehoek: terug naar het fundament.

Rosa

Hackers United — zelf hosten & digitale soevereiniteit

Zelf hosten Servers Static site generators Digitale afhankelijkheid Planned obsolescence

Wat zij vertelde

Rosa — van Hackers United — stelde een vraag die ik me nog nooit had gesteld: waar leeft je website eigenlijk? Het antwoord is bijna altijd "de cloud" of "AWS" — wat in de praktijk betekent: op servers van een paar grote bedrijven, ver buiten jouw controle.

Wat is een hacker?

Niet wat films ervan maken. In de context van Rosa: iemand die creatief omgaat met systemen en beperkingen. Voorbeelden: een backend draaien op Google Sheets, je eigen CMS bouwen, een proof of concept maken in Keynote.

Planned obsolescence

Apparaten en software worden bewust ontworpen om te verouderen — plannen werken reparatie en hacken actief tegen. De enige oplossing lijkt soms "niet kopen" en meer afhankelijkheid vermijden.

Rosa the travelling server

Rosa heeft een fysieke server die ze overal mee naartoe neemt. Een server is gewoon een computer. Oudere hardware dwingt je anders na te denken over frameworks en keuzes in HTML/CSS.

Static site generators

Minder belasting voor de server. Als alles faalt: je hebt nog steeds markdown. Leesbare code. Eleventy (11ty) als voorbeeld: grote community, veel documentatie.

Zelf hosten

Web dev server met Linux en Yunohost. Zelf hosten van: messaging platform, je eigen website, een radioshow. Technisch mogelijk, maar vraagt bewuste keuze.

Wat ik heb meegenomen

Ik had nog nooit nagedacht over waar mijn websites echt leven. Rosa maakte dat concreet. Het is niet alleen een technische keuze — het is een politieke keuze. En het koppelt direct aan wat Marleen Stikker later zou zeggen over digitale afhankelijkheid.

Johan Huijkman

Accessibility & WCAG in de praktijk

WCAG 2.2 POUR principes Screenreader Slechtziend Autisme Mobiliteitsbeperking

Wat hij vertelde

Johan Huijkman gaf een van de meest concrete sessies van de minor. Hij opende met een quote die me bijbleef:

"Je gaat het pad zien als je het doorhebt. En je hoort het pad als je luistert."

WCAG staat voor Web Content Accessibility Guidelines. De vier basisprincipes — POUR — zijn de fundering van alles: Perceivable (waarneembaar), Operable (bedienbaar), Understandable (begrijpelijk) en Robust (robuust).

Drie gebruikers, drie problemen

Martijn

Slechtziend. Geen schermbeveiliging, maar alles heel groot zoomen. Door elementen die in en uitschuiven verspringt de layout bij 400% zoom — waardoor delen van de pagina onbereikbaar worden. Wat helpt: gerelateerde info bij elkaar, animaties uitzetten, schaalbaarheid, focusrand.

Naduah

Mobiliteitsproblemen. Problemen op de site: te veel velle kleuren, te veel info tegelijk, te veel klikken voor info, info ver uit elkaar. Wat helpt: rustige layout, minder stappen, info groeperen.

Guido

Autist. Kan visuele en auditieve prikkels niet filteren. Wat helpt: rustige layout en vormgeving, duidelijke instructies en meldingen, voorspelbaarheid — geen verrassende animaties of pop-ups.

Johan liet ook zien hoe je met "H" door headings kunt navigeren in een screenreader. Elke foto hoeft geen alt-tekst — maar wél een leeg alt="" attribuut, zodat de screenreader de afbeelding overslaat.

Wat ik heb meegenomen

Deze sessie sloot perfect aan op het HCD-vak. Berend, Martijn, Naduah, Guido — ze zijn allemaal anders. Accessibility is geen checklist die je afvinkt; het is begrijpen hoe iemand de wereld ervaart en daar je ontwerp op aanpassen. Testen met 400% zoom had ik daarvoor nooit gedaan. Nu altijd.

Marleen Stikker

The web you want — open social web

Open social web Fediverse ActivityPub Privacy by design Big Tech regulatie Digitale soevereiniteit

Wat zij vertelde

Marleen Stikker van Waag Futurelab sloot de Weekly Nerd-reeks af met een van de meest urgente sessies: het web zoals we het willen versus het web zoals het nu is. Sociale media bestaan al van 1994 — maar we hebben nooit de controle gehad over onze privacy of wat we zien via algoritmes.

"Waarom moet een stofzuiger foto's sturen naar het bedrijf?"

— Marleen Stikker

Dat is geen rare vraag. Het is een symptoom van een groter probleem: digitale afhankelijkheid en surveillance by design. Elk apparaat, elk platform — gebouwd om data te verzamelen, niet om je te dienen.

Protocol, not platform

Het internet is gebouwd op protocollen (open standaarden). Sociale media zijn gesloten platforms. Het Fediverse keert dit om: open protocollen (ActivityPub) waarop iedereen een eigen server kan draaien.

Fediverse

Gefedereerd netwerk van verbonden servers. X → Mastodon, Instagram → Pixelfed, YouTube → PeerTube, TikTok → Loops.video. Geen centrale eigenaar, geen algoritme dat je gedrag verkoopt.

Regulatie als voorwaarde

EU-wetgeving (GDPR, Digital Services Act, Data Governance Act, AI Act) opent markten, stimuleert creativiteit, verwelkomt nieuwe spelers en verdedigt democratie. Regulatie is niet de vijand van innovatie.

Public stack

Nee: locked-in, AI-monopolies, surveillance, centralisatie, extractief. Ja: interoperabel, public AI, privacy by design, gedistribueerd, regeneratief, publieke waarden, democratie in control.

Campagnes die Marleen noemde: #MakeSocialsSocialAgain, People vs Big Tech, Public Spaces, Free Our Feeds.

Wat ik heb meegenomen

Deze sessie verbond alles: Rosa's vraag over waar je website leeft, PPK's uitleg over hoe browsers werken als publiek platform, de Weekly Geek sessie over agentic AI. Het web is niet neutraal — het is ontworpen door mensen met belangen. Als developer kies ik elke dag mee hoe dat web eruitziet. Dat is een verantwoordelijkheid die ik daarvoor onderschatte.
