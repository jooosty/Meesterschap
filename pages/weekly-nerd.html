Killian Valkhof

CSS as a replacement for JavaScript

Rule of Least Power CSS-only interactie Backward compatibility Input styling Accordeons & dialogs

Wat hij vertelde

Killian opende met een principe dat me de hele minor is bijgebleven: de Rule of Least Power kies altijd de minst krachtige taal die geschikt is voor een taak. Voor het web betekent dat: kijk eerst naar HTML, dan CSS, dan pas JavaScript.

"Je kan meer CSS gebruiken voor dingen dan je denkt, in plaats van JS."

Browsers zijn backwards compatible als iets ooit heeft gewerkt, moet het altijd blijven werken. Dat is een belofte die browsers aan het web doen. Killian liet zien dat je dat als voordeel kunt gebruiken: CSS die je leert, hoef je nooit opnieuw te leren.

appearance: none

Haalt alle browser-styling weg van een input, zodat je hem volledig zelf kunt stylen. Inputs waren 10 jaar geleden moeilijk te stylen nu niet meer.

scroll-behaviour: smooth

Smooth scrollen in puur CSS, zonder JavaScript. De browser bepaalt de animatie en respecteert automatisch prefers-reduced-motion.

scroll-margin

Als je een fixed header hebt en smooth scrollt naar een anker, zorgt scroll-margin dat de pagina stopt met de juiste afstand tot de top.

<details> & accordeons

Native accordeon zonder JavaScript. Het name attribuut zorgt dat maar één tegelijk open kan staan. ::marker overschrijft het pijltje. <details open> forceert de open staat.

<dialog>

De moderne vervanger van alert(). Je kunt hem stylen alert niet. Standaard onzichtbaar, needs JS om te openen maar de rendering en focus-trap zijn gratis.

prefers-reduced-motion

Gebruikers kunnen in het OS aangeven dat ze minder beweging willen. @media (prefers-reduced-motion: reduce) respecteert dit in CSS.

Een scherpe observatie: als je iets leert, check dan niet alleen of het werkt maar of er iets beter bestaat. Kennis is geen eindbestemming.

Wat ik heb meegenomen

De Rule of Least Power herkende ik direct in KRAFTWERK (CSS to the Rescue). Maar het echte inzicht was breder: elke keer als ik nu JavaScript schrijf, vraag ik me eerst af of HTML of CSS het ook kan. Vaak wel.

Peter-Paul Koch

Browsers & support detection

Browser architectuur Rendering engine Render blocking Backward compatibility Browsergeschiedenis

Wat hij vertelde

Peter-Paul Koch auteur van quirksmode.org en een van de meest ervaren browser-experts ter wereld gaf een lezing over hoe browsers eigenlijk werken en hoe we daarmee moeten omgaan als developers.

Wat zit er in een browser?

Rendering engine

Verantwoordelijk voor HTML en CSS parsing, rendering, en de constructie van de DOM en AOM-trees. Niet voor de interface of JavaScript engine. Chrome gebruikt Blink (afgesplitst van WebKit).

AOM Accessibility Object Model

Naast de DOM bouwt de browser ook een AOM een parallelle boom die screenreaders en andere assistive technology gebruiken om de pagina te begrijpen.

Core vs DOM

Core: wat JavaScript een programmeertaal maakt (variabelen, functies, loops). DOM: de manier waarop JS delen van een HTML-pagina kan aanspreken en wijzigen. Twee aparte dingen.

Render blocking

CSS blokkeert altijd rendering. JavaScript ook tenzij je async of defer gebruikt. Defer: laad later, voer uit als DOM klaar is. Async: laad en voer uit zodra klaar, ongeacht DOM.

Een korte browsergeschiedenis

PPK nam ons mee door de browseroorlog en hoe die het web heeft gevormd:

1990: WorldWideWeb de allereerste browser, gebouwd door Tim Berners-Lee. 1993: Mosaic eerste browser voor gewone mensen. 1994: Netscape eerste betaalde browser. 1995: Internet Explorer & Opera (Opera's oprichter was ook uitvinder van CSS). 2000: Konqueror hobbyproject van Linux-gebruikers, engine later opgekocht door Apple. 2003: Safari. 2004: Firefox. 2007: Safari iOS. 2008: Chrome met de Blink-engine. 2024: Ladybird nieuwe rendering engine, gestart als hobbyproject.

"Browsers are the most hostile development platforms in the world."

— Douglas Crockford (maker van JSON)

Backward compatibility: als iets ooit heeft gewerkt in een browser, moet het altijd blijven werken. Geen doctype in je HTML? Dan valt de browser terug op Internet Explorer 6-gedrag.

Wat ik heb meegenomen

Ik kende het verschil tussen DOM en Core niet echt ik gebruikte beide gewoon als "JavaScript". Het AOM-model was nieuw voor mij en sloot direct aan op het HCD-vak: de browser bouwt een aparte boom voor screenreaders. Als je die negeert in je HTML, negeert de screenreader jouw content.

Nils Binder

9Elements · Design voor het web

Figma vs Photoshop Web development triangle Design tools CSS Grid live

Wat hij vertelde

Nils Binder werkt bij 9Elements een bureau dat communications design, product development en web development combineert. Ze werkten onder andere voor Schalke 04 en de OECD.

Hij nam ons mee door hoe het designproces voor het web is veranderd: van Photoshop (20 jaar geleden) naar Figma (nu). Photoshop was gemaakt voor printontwerp en deed dingen die op het web helemaal niet mogelijk waren. Figma is gebouwd voor het web en sluit veel beter aan op wat browsers kunnen.

Figma vs Photoshop

Photoshop: overcompliceerd, gemaakt voor print, makkelijk dingen ontwerpen die het web niet ondersteunt. Figma: eenvoudiger, variabelen, dichter bij hoe het web werkt.

Web development triangle

De huidige workflow bij veel bureaus: Figma → Tailwind → AI. Nils liet zien hoe dat de drempel verlaagt maar ook risico's meebrengt voor kwaliteit en begrip.

CSS Grid live

Nils demonstreerde live hoe je een nieuwspagina-layout bouwt met grid-template-columns: 2fr 3fr auto 1fr hoe weinig code je nodig hebt als je CSS Grid echt begrijpt.

"In Figma is alles mogelijk wat op het web kan. Maar niet alles wat op het web kan, is beschikbaar in Figma."

Wat ik heb meegenomen

De web development triangle (Figma → Tailwind → AI) zette me aan het denken. Het maakt dingen sneller, maar het vergroot ook de kans dat je nooit echt begrijpt waarom iets werkt. De minor is een antwoord op die driehoek: terug naar het fundament.

Rosa

Hackers United zelf hosten & digitale soevereiniteit

Zelf hosten Servers Static site generators Digitale afhankelijkheid Planned obsolescence

Wat zij vertelde

Rosa van Hackers United stelde een vraag die ik me nog nooit had gesteld: waar leeft je website eigenlijk? Het antwoord is bijna altijd "de cloud" of "AWS" wat in de praktijk betekent: op servers van een paar grote bedrijven, ver buiten jouw controle.

Wat is een hacker?

Niet wat films ervan maken. In de context van Rosa: iemand die creatief omgaat met systemen en beperkingen. Voorbeelden: een backend draaien op Google Sheets, je eigen CMS bouwen, een proof of concept maken in Keynote.

Planned obsolescence

Apparaten en software worden bewust ontworpen om te verouderen plannen werken reparatie en hacken actief tegen. De enige oplossing lijkt soms "niet kopen" en meer afhankelijkheid vermijden.

Rosa the travelling server

Rosa heeft een fysieke server die ze overal mee naartoe neemt. Een server is gewoon een computer. Oudere hardware dwingt je anders na te denken over frameworks en keuzes in HTML/CSS.

Static site generators

Minder belasting voor de server. Als alles faalt: je hebt nog steeds markdown. Leesbare code. Eleventy (11ty) als voorbeeld: grote community, veel documentatie.

Zelf hosten

Web dev server met Linux en Yunohost. Zelf hosten van: messaging platform, je eigen website, een radioshow. Technisch mogelijk, maar vraagt bewuste keuze.

Wat ik heb meegenomen

Ik had nog nooit nagedacht over waar mijn websites echt leven. Rosa maakte dat concreet. Het is niet alleen een technische keuze het is een politieke keuze. En het koppelt direct aan wat Marleen Stikker later zou zeggen over digitale afhankelijkheid.

Johan Huijkman

Accessibility & WCAG in de praktijk

WCAG 2.2 POUR principes Screenreader Slechtziend Autisme Mobiliteitsbeperking

Wat hij vertelde

Johan Huijkman gaf een van de meest concrete sessies van de minor. Hij opende met een quote die me bijbleef:

"Je gaat het pad zien als je het doorhebt. En je hoort het pad als je luistert."

WCAG staat voor Web Content Accessibility Guidelines. De vier basisprincipes POUR zijn de fundering van alles: Perceivable (waarneembaar), Operable (bedienbaar), Understandable (begrijpelijk) en Robust (robuust).

Drie gebruikers, drie problemen

Martijn

Slechtziend. Geen schermbeveiliging, maar alles heel groot zoomen. Door elementen die in en uitschuiven verspringt de layout bij 400% zoom waardoor delen van de pagina onbereikbaar worden. Wat helpt: gerelateerde info bij elkaar, animaties uitzetten, schaalbaarheid, focusrand.

Naduah

Mobiliteitsproblemen. Problemen op de site: te veel velle kleuren, te veel info tegelijk, te veel klikken voor info, info ver uit elkaar. Wat helpt: rustige layout, minder stappen, info groeperen.

Guido

Autist. Kan visuele en auditieve prikkels niet filteren. Wat helpt: rustige layout en vormgeving, duidelijke instructies en meldingen, voorspelbaarheid geen verrassende animaties of pop-ups.

Johan liet ook zien hoe je met "H" door headings kunt navigeren in een screenreader. Elke foto hoeft geen alt-tekst maar wél een leeg alt="" attribuut, zodat de screenreader de afbeelding overslaat.

Wat ik heb meegenomen

Deze sessie sloot perfect aan op het HCD-vak. Berend, Martijn, Naduah, Guido ze zijn allemaal anders. Accessibility is geen checklist die je afvinkt; het is begrijpen hoe iemand de wereld ervaart en daar je ontwerp op aanpassen. Testen met 400% zoom had ik daarvoor nooit gedaan. Nu altijd.

Marleen Stikker

The web you want open social web

Open social web Fediverse ActivityPub Privacy by design Big Tech regulatie Digitale soevereiniteit

Wat zij vertelde

Marleen Stikker van Waag Futurelab sloot de Weekly Nerd-reeks af met een van de meest urgente sessies: het web zoals we het willen versus het web zoals het nu is. Sociale media bestaan al van 1994 maar we hebben nooit de controle gehad over onze privacy of wat we zien via algoritmes.

"Waarom moet een stofzuiger foto's sturen naar het bedrijf?"

— Marleen Stikker

Dat is geen rare vraag. Het is een symptoom van een groter probleem: digitale afhankelijkheid en surveillance by design. Elk apparaat, elk platform gebouwd om data te verzamelen, niet om je te dienen.

Protocol, not platform

Het internet is gebouwd op protocollen (open standaarden). Sociale media zijn gesloten platforms. Het Fediverse keert dit om: open protocollen (ActivityPub) waarop iedereen een eigen server kan draaien.

Fediverse

Gefedereerd netwerk van verbonden servers. X → Mastodon, Instagram → Pixelfed, YouTube → PeerTube, TikTok → Loops.video. Geen centrale eigenaar, geen algoritme dat je gedrag verkoopt.

Regulatie als voorwaarde

EU-wetgeving (GDPR, Digital Services Act, Data Governance Act, AI Act) opent markten, stimuleert creativiteit, verwelkomt nieuwe spelers en verdedigt democratie. Regulatie is niet de vijand van innovatie.

Public stack

Nee: locked-in, AI-monopolies, surveillance, centralisatie, extractief. Ja: interoperabel, public AI, privacy by design, gedistribueerd, regeneratief, publieke waarden, democratie in control.

Campagnes die Marleen noemde: #MakeSocialsSocialAgain, People vs Big Tech, Public Spaces, Free Our Feeds.

Wat ik heb meegenomen

Deze sessie verbond alles: Rosa's vraag over waar je website leeft, PPK's uitleg over hoe browsers werken als publiek platform, de Weekly Geek sessie over agentic AI. Het web is niet neutraal het is ontworpen door mensen met belangen. Als developer kies ik elke dag mee hoe dat web eruitziet. Dat is een verantwoordelijkheid die ik daarvoor onderschatte.

Hans de Zwart

Niets te verbergen? — Of toch wel?

Privacy AVG / GDPR Contextuele integriteit Dataportabiliteit Agentic AI Surveillance

Wat hij vertelde

Hans de Zwart opende met de vraag die iedereen denkt te kunnen beantwoorden maar dat niet kan: wat is privacy eigenlijk? Hij legde drie lagen bloot. Privacy als individueel recht — met rust gelaten worden. Als sociaal recht — in elke context iemand anders kunnen zijn. En als democratische noodzaak — de ruimte om je mening autonoom te vormen, zonder dat iemand meekijkt.

"Ik loop door een wereld waar ik niets wil kopen, maar ik zit in een wereld die me alles probeert aan te smeren."

— Hans de Zwart

Het concept dat me het meest bijbleef is contextuele integriteit: privacy gaat niet over geheimhouding, maar over verwachting. Je deelt andere dingen met je dokter dan met je werkgever, en dat klopt. Het wordt pas een probleem als gegevens buiten die verwachte context terechtkomen — bij partijen die je er niet voor had uitgenodigd. Eigenlijk is dat gewoon gezond verstand over hoe je met elkaar omgaat, toegepast op data.

Hoe de wet het regelt

Toestemming

De meest gebruikte juridische grondslag om persoonsgegevens te verwerken. Maar toestemming moet vrij én geïnformeerd zijn, en moet altijd ingetrokken kunnen worden.

Doelbinding

Je mag data alleen gebruiken voor het doel waarvoor je die hebt verzameld. Toestemming voor doel 1 is géén toestemming voor doel 2. Klinkt logisch, maar wordt massaal genegeerd.

Rechten van betrokkenen

Inzage, correctie, aanvulling. Het recht op vergetelheid. En dataportabiliteit: je mag je gegevens meenemen naar een andere partij. Die rechten bestaan al — ze worden alleen zelden opgeëist.

Privacy by design

Minimaliseren: sla geen gegevens op die je niet nodig hebt. Wat je niet hebt, kan niet lekken. Slimme cryptografie, hashes, differential privacy — je kunt vaak de gewenste functionaliteit bouwen zónder de volledige data.

Het web als surveillance-machine

Hans stelde een simpele vraag: als je met een gloednieuwe browser naar nu.nl gaat, hoeveel partijen weten dan van dat bezoek? Het antwoord is schrikbarend. In real time worden gegevens doorgestuurd naar tientallen adverteerders, nog voor de pagina volledig geladen is. En dan is er nog een tweede vraag: bestaat er nog een websitebezoek dat niet bij Google in beeld is?

En dan is er AI. ChatGPT is getraind op het internet — dus op alles wat er over jou online staat. Agentic AI maakt het nog urgenter: agents gaan steeds dieper je computer in, koppelen tools aan elkaar, handelen namens jou. Dat maakt een nieuw soort surveillance mogelijk, één waarbij data niet alleen wordt bewaard maar ook actief wordt ingezet.

Wat ik heb meegenomen

Deze sessie sloot bijna te goed aan op Marleen Stikker. Waar zij het had over het web dat je wilt, had Hans het over de prijs die je betaalt voor het web dat er nu is. Wat me raak: als developer beslis ik mee. Welke data vraag ik op? Welke sla ik op? Doelbinding is geen juridische formaliteit — het is een ontwerpbeslissing. Minimaliseren is ook een ontwerpbeslissing. Dat zijn mijn keuzes.

Marvin Sernee — Q42

Design Systems & Efficiëntie

Design systems Atomic Design Design tokens Accessibility AI in dev Stakeholder management

Wat hij vertelde

Marvin Sernee werkt bij Q42, een nerdagency die digitale producten bouwt voor grote organisaties. Zijn sessie ging over design systems — niet als theoretisch concept maar als iets wat je in een bestaande organisatie moet zien te laten landen, met alle weerstand en legacy die daarbij komen kijken.

Elk traject begint met een assessment. Waarom werkt de huidige situatie niet? Waar wil je naartoe? Waar sta je nu? Wat heb je nodig om er te komen? Dat klinkt simpel, maar die vragen scherp beantwoorden is al het halve werk. Vanuit die basis bouw je een design language, een Figma-structuur en een schaalbaar system — en vertaal je dat naar een roadmap.

Waarom Atomic Design niet werkt

Marvin was eerlijk over Atomic Design: in theorie mooi, in de praktijk lastig. De onderdelen worden al snel te klein en te specifiek, waardoor niemand het meer eens is over wat nou precies een atom, molecule of organism is. Ze kozen voor een ander mentaal model.

Foundation

De basisbouwstenen: design tokens, iconen, illustraties, basale styling. Witruimte, kleuren, border radius. Alles wat de rest draagt.

Core components

Zelfstandige, herbruikbare componenten. Buttons, inputs, cards. Dingen die op zichzelf staan en overal inzetbaar zijn.

Herbruikbare patronen

Combinaties van meerdere componenten die samen een gebruikersflow of interactie oplossen. Een stap hoger dan losse componenten.

Nooit af

Een design system is geen project met een eindpunt. Het groeit mee met het product, de organisatie en de gebruikers. Stoppen is verslechteren.

Werken voor grote klanten

Bij grote organisaties spelen er meer krachten. Meer stakeholders, meer legacy, vaste brand guidelines, meer afhankelijkheden. Business stakeholders denken vanuit hun wensen, niet altijd vanuit de technische consequenties. En iedereen heeft de hoogste prioriteit.

"Gedeelde verantwoordelijkheid is geen verantwoordelijkheid."

— Marvin Sernee

Zijn advies: focus altijd op het probleem, niet op de voorgestelde oplossing. Stakeholders komen met oplossingen; jouw taak is te begrijpen welk probleem erachter zit. De impact is groot — verbeteringen bij een grote organisatie bereiken direct een grote groep bestaande gebruikers.

AI als junior, niet als vervanger

Marvin had een heldere kijk op AI die ik nergens anders zo geformuleerd had gehoord: behandel AI als een peuter. Geef duidelijke richtlijnen, voorbeelden, context. Hoe beter de instructies, hoe beter de output. Niet als excuus om lui te worden — maar als versnelling voor het werk dat je sowieso al leuk vindt. Debuggen, codebases analyseren, repetitieve taken: prima. Creatief werk dat je vakmanschap vraagt: doe het zelf.

Toegankelijkheid zonder excuses

Het laatste deel van zijn sessie ging over accessibility bij grote klanten. De drempel zo laag mogelijk maken: werksessies met designers én developers, vroeg in het proces, niet als losse audit aan het einde. Als je toegankelijkheid vanaf het begin meeneemt, kost het nauwelijks extra tijd.

Zijn punt over het native <dialog>-element sloot perfect aan op wat Killian eerder vertelde: de browser biedt al een toegankelijke oplossing met screenreader-ondersteuning en focusbeheer ingebouwd. Waarom zelf iets bouwen dat slechter is?

Wat ik heb meegenomen

Het model Foundation → Core components → Patronen is iets wat ik direct kan toepassen, ook op kleinere projecten. Maar het echte inzicht was het verhaal over stakeholders: focus op het probleem, niet de oplossing. En de uitspraak over gedeelde verantwoordelijkheid bleef hangen — die gaat over meer dan design systems.

De Voorhoede

Marleen · Peter · Bas — drie invalshoeken, één bureau

Headless CMS mix-blend-mode NL Design System Design tokens Offline-first Accessibility

Wat zij vertelden

De Voorhoede stuurde drie mensen — Marleen, Peter en Bas — die elk een ander stuk van het vak belichten. Zelden zo'n sessie gehad waarbij drie heel verschillende perspectieven zo goed op elkaar aansloten.

Marleen

Medior front-end developer. Werkt veel met DatoCMS, een headless CMS waarbij de front-end volledig losstaat van het contentbeheer. De klant beheert zelf de content; het team bouwt de presentatielaag. Een groot deel van het dagelijks werk bestaat uit het oplossen van CMS-gerelateerde problemen: custom classes schrijven, plugins bouwen, blogfunctionaliteiten implementeren. Niet altijd glamoureus, maar het houdt producten draaiend. Haar voorbeeld van mix-blend-mode als creatieve CSS-oplossing was een goede herinnering dat je met standaard eigenschappen verder kunt komen dan je denkt.

Peter

Senior front-end developer, gespecialiseerd in accessibility. Vertelde over het implementeren van het NL Design System — een bestaande site opnieuw opbouwen met herbruikbare componenten in Astro. Centraal staat het idee van design tokens: CSS opgesplitst in twee lagen. Eerst de waarden zelf — kleuren, spacing, border radius — in een tokens.json. Dan de plekken waar die waarden worden toegepast. Consistent, onderhoudbaar, en makkelijk aan te passen zonder door de hele codebase te zoeken. Voor accessibility: focusindicator, toetsenbordnavigatie, screenreader-ondersteuning. Niet als afterthought, maar als startpunt.

Bas

Offline-first development. Liet een webapplicatie voor liftinspecteurs zien die ook zonder internet moet werken. Zijn analyse van waarom projecten uitlopen was raak: klanten weten aan het begin nog niet precies wat ze willen. Nieuwe inzichten ontstaan tijdens het bouwen, scope groeit mee. Bij offline-first mag je dat niet als excuus gebruiken — je moet vanaf dag één nadenken over wat er gebeurt als er geen verbinding is. Offline-ondersteuning is geen extra functie die je er achteraf bijplakt. Het is een uitgangspunt.

"Offline ondersteuning is geen extra functie — het is een uitgangspunt van het ontwerp."

— Bas, De Voorhoede

Wat ik heb meegenomen

Peters tokens-opzet — waarden apart van toepassingen — is precies wat ik in dit project ook heb gedaan met tokens.css en themes.css. Fijn om te horen dat dat een bewuste keuze is bij een productioneel bureau, niet alleen een persoonlijke voorkeur. En het verhaal van Bas over scope-creep herkende ik direct: offline-first dwingt je eerlijk te zijn over wat een applicatie echt nodig heeft. Dat is een goede les voor elk project.

Cyd Stumpel

Prefers enhanced motion

Progressive enhancement Graceful degradation prefers-reduced-motion interpolate-size Popover API CSS-only animaties

Wat zij vertelde

Cyd Stumpel opende met haar versie van het web dat ze wil: vol beweging, creativiteit en openheid. Het open web is wat haar het meest aantrekt aan het vak — tegelijkertijd is die openheid ook kwetsbaar. Websites zijn makkelijk te kopiëren, en AI maakt dat alleen maar makkelijker.

Maar de echte scherpe observatie kwam daarna. Onderdelen die Cyd zelf voor het web heeft ontworpen, bleken ontoegankelijk te zijn voor mensen zoals Léonie — een blinde screenreader-gebruiker. Dat confronteert. Het web als gereedschap werkt alleen als iedereen het kan gebruiken.

"When you deliberately design for the worst acceptable experience first, everything that comes after becomes an enhancement instead of a requirement."

— Cyd Stumpel

Dat is de kern van progressive enhancement. Haar analogie: een roltrap kan kapotgaan, maar je kunt hem nog steeds als trap gebruiken. Graceful degradation betekent dat animaties en visuele effecten de kerntaak nooit ondermijnen. Begin met wat altijd moet werken. Alles daarna is een verbetering, geen vereiste.

HTML first, CSS als laag

Cyd liet zien hoeveel er de afgelopen jaren aan HTML is toegevoegd zonder dat veel mensen het door hebben. Het <details>-element werkt al zonder JavaScript — en met het name-attribuut kun je een exclusieve accordeon bouwen waarbij maar één item tegelijk open staat: 

<details name="faq">…</details>
<details name="faq">…</details>

De uitdaging is dat CSS-verbeteringen bovenop die semantiek een gelaagde aanpak vragen. Haar voorbeeld met interpolate-size: allow-keywords laat zien hoe dat werkt: animeren van height: 0 naar height: auto was jarenlang onmogelijk in CSS. Nu kan het — maar alleen in Firefox en Chrome, en alleen als je het correct omhult met @supports en @media (prefers-reduced-motion: no-preference): 

@supports (interpolate-size: allow-keywords) {
  @media (prefers-reduced-motion: no-preference) {
    ::details-content {
      height: 0;
      overflow: clip;
      transition: height 0.4s ease-out, content-visibility, allow-discrete;
    }
    details[open]::details-content {
      height: auto;
    }
  }
}

Popover API zonder JavaScript

Het meest indrukwekkende stuk was de Popover API. Een popupmenu met animatie, puur in CSS — geen JavaScript nodig. De truc zit in transition-behavior: allow-discrete en @starting-style, waarmee je de beginstaat van een animatie kunt definiëren voordat een element zichtbaar wordt: 

nav {
  @supports selector(:popover-open) {
    @media (prefers-reduced-motion: no-preference) {
      clip-path: inset(0 100% 100% 0 round 0.15rem);
      transition: 2s ease-out;
      transition-behavior: allow-discrete;

      &:popover-open {
        clip-path: inset(0 round 0.15rem);

        @starting-style {
          clip-path: inset(0 100% 100% 0 round 0.15rem);
        }
      }
    }
  }
}

Wat ik heb meegenomen

De volgorde die Cyd beschrijft — HTML eerst, dan CSS als laag, dan animatie als verbetering — is eigenlijk wat Killian ook bedoelde met de Rule of Least Power. Maar Cyd maakt het concreter: het gaat niet alleen om welke taal je kiest, maar ook om de volgorde van je beslissingen. Eerst de kerntaak. Dan de laag. Dan de glitter. En @starting-style kende ik nog niet — dat ga ik direct uitproberen.
