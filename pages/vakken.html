CSS to the
Rescue

Een volledig functioneel kerncentrale bedieningspaneel gebouwd met bijna uitsluitend CSS. Geen JavaScript voor interactie, geen ID- of class-selectors voor styling — alleen de kracht van CSS.

De opdracht

Maak een interactief component met bijna uitsluitend CSS. De twee kernregels: geen IDs of classes als styling-hooks (alleen voor :target en <label for>), en geen JavaScript voor logica — alleen een minimale range-to-custom-property bridge was toegestaan.

Het doel: de grenzen van CSS opzoeken. Alles wat je normaal met JavaScript zou doen, moet opgelost worden met selectors, pseudo-classes en de cascade.

Het concept: KRAFTWERK

Ik koos voor een fictief Duits kerncentrale bedieningspaneel, geïnspireerd op fotografieën van verlaten Oost-Europese controlekamers. De naam KRAFTWERK — Duits voor "energiecentrale" — paste perfect bij de retro-industriële esthetiek.

Het paneel bevat vier analoge meters, zes LED-indicatoren, drie toggleschakelaars, vier draaiknoppen, START/STOP besturing en een retro scroll-drum MW display. Alle interacties zijn puur CSS — ze zijn onderling verbonden: schakel je een pomp uit terwijl de centrale draait, dan flash het schakelpaneel oranje.

Componenten

Analoge meters × 4

Frequency, Pressure, Temperature, Power. Naald rotatie aangestuurd via CSS custom property van de knob-range input. Werkt alleen als de centrale actief is.

Toggle switches × 3

Pump A, Pump B, Turbine. Pure input[type=checkbox] met gestylede label. Schakelen ze uit tijdens het draaien, dan activeert een oranje waarschuwingsflash.

Draaiknoppen × 4

input[type=range] gestyled als roterende knop. JS schrijft alleen de knob-waarde naar een CSS custom property. Rotatie, kleur en alle feedback: puur CSS.

LED indicatoren × 6

Reageren op de toggle switches: zet je Pump A uit, gaan de eerste twee LEDs donker. Alles via de ~ sibling combinator en :not(:checked).

START / STOP

Anchor links die :target activeren op verborgen <div> elementen buiten <main>. De hele interactieve staat van het paneel hangt hier aan.

MW Scroll display

Retro "drum"-display gebouwd met 21 <span>-elementen en een scroll-animatie. Dimmed als de centrale uit staat, flikkert rood bij een meltdown.

EMERGENCY meltdown

Alle vier knoppen op maximum + centrale actief = EMERGENCY overlay. Volledig via container style queries. Het scherm schudt, de display flikkert rood.

Easter eggs × 5

Inspectie-stempel, ingenieursgravure, koffievlek, versleten pomp-label, en een plantnaam die verschijnt bij 2 seconden hoveren op de titel.

Code highlight: container style queries voor meltdown

Het gevaarlijkste onderdeel: als alle vier knoppen op 145 staan én de centrale draait, treedt er een noodtoestand op. Dit vereist het checken van vier CSS custom property waarden tegelijk — iets wat normaal JavaScript vereist. De oplossing: geneste container style queries: 

/* state.css — meltdown triggert alleen als alle 4 knoppen op 145 staan */
@container style(--knob-power: 145) {
  @container style(--knob-freq: 145) {
    @container style(--knob-pressure: 145) {
      @container style(--knob-temp: 145) {

        #running:target ~ main > p:last-of-type {
          opacity: 1;
          animation: emergency-pulse 0.8s ease-in-out infinite;
        }

        #running:target ~ main {
          animation:
            meltdown-shake 0.15s ease-in-out infinite,
            overload-flash  0.6s ease-in-out infinite;
        }

      }
    }
  }
}

De knoppen schrijven hun waarde via één regel JavaScript naar :root als custom property. Al het rest — het lezen, vergelijken en reageren — is puur CSS.

Code highlight: letter-corrupt animatie op de titel

De letters van KRAFTWERK flikkeren onregelmatig en wisselen kort van glyph — alsof er een elektrische storing is. Elke letter heeft zijn eigen timing via per-letter CSS variabelen: 

/* easter-eggs.css — elke letter krijgt eigen timing */
/* K */ header > h1 > span:first-child > span:nth-child(1) {
  --glyph:            'К';        /* Cyrillische lookalike */
  --corrupt-duration: 11.3s;
  --corrupt-delay:    1.2s;
  --flicker-duration: 2.8s;
}

header > h1 > span > span {
  animation:
    letter-stamp-in  0.4s cubic-bezier(0.22, 1, 0.36, 1)
                     var(--stamp-delay) forwards,
    letter-flicker   var(--flicker-duration) var(--flicker-delay)
                     ease-in-out infinite,
    letter-hide      var(--corrupt-duration) var(--corrupt-delay)
                     steps(1, end) infinite;
}

De letter-hide animatie maakt de echte letter voor ~100ms transparant precies op het moment dat de ::before pseudo-element de vervangings-glyph toont — zodat ze nooit tegelijk zichtbaar zijn.

Code highlight: knob → meter binding via custom properties

De enige JavaScript in het hele project schrijft de waarde van een input[type=range] naar een CSS custom property op :root. CSS pakt die waarde op en berekent de rotatie van de naald: 

/* Het enige JS in het project — range → custom property bridge */
const updateRange = range => {
  document.documentElement.style.setProperty(
    '--' + range.name,
    range.value
  );
};

/* state.css — CSS pakt de waarde op, berekent rotatie */
#running:target ~ main > section:first-of-type
  > figure:nth-of-type(1) > div > span:nth-child(3) {
  transform: translateX(-50%)
    rotate(calc(var(--knob-freq) * 0.48deg));
}

Experimenten die niet werkten

Drie dingen die ik probeerde, maar waar CSS fundamenteel niet voor geschikt is — en dat leerde me juist meer over de grenzen van het medium:

× Geluidseffecten

Ik wilde kliksgeluiden bij de schakelaars en een alarmtoon bij de meltdown. CSS heeft geen audio capabilities — dit is fundamenteel onmogelijk zonder JS.

× CSS-only timer

Een timer die bijhoudt hoe lang de centrale al draait. Er bestaat een CSS-timer truc, maar deze integreren met de :target state van het paneel bleek niet correct werkbaar.

× Willekeurige startwaarden

De knoppen beginnen altijd op dezelfde startpositie. CSS heeft geen random-functie die beginwaarden van range inputs kan instellen — daarvoor is JS vereist.

Nieuwe inzichten

01

CSS kan meer interactiviteit dan je denkt

Draaiknoppen, conditional waarschuwingen, een meltdown — allemaal zonder logica-JS. De sleutel: denk in termen van state via inputs en pseudo-classes, niet in termen van event listeners.

02

Container style queries = CSS conditional logic

Door custom property waarden te lezen via @container style() kun je CSS conditioneel maken op basis van data — niet alleen op basis van afmetingen. Dit is krachtig en nog relatief onbekend.

03

Selector specificity als architectuurtool

Zonder classes moet je bewust navigeren met sibling combinators (~), :nth-of-type() en :not(). Dit dwingt je om de HTML-structuur als ontwerpdocument te behandelen.

04

Kleurnotaties voor dynamische gradients

hsl(), oklch(), hex — elke notatie heeft voordelen voor bepaalde use cases. hsl() maakt het makkelijk om tonen en tinten programmatisch te berekenen via calc().

05

@layer voor cascade-controle

Door CSS in lagen te organiseren kun je de cascade expliciet sturen zonder te vertrouwen op specificity of volgorde van @import. Maakt grote CSS-codebases aanzienlijk beheersbaarder.

06

CSS grenzen bewust kennen

Het uitproberen van audio, timers en random — en zien waarom die falen — leerde me meer over CSS dan de dingen die wel werkten. De grenzen kennen is net zo waardevol als de mogelijkheden.

Reflectie

CSS to the Rescue was het vak waarbij ik het meest verrast werd — door CSS zelf. Ik wist dat je er visuele dingen mee kon maken, maar een volledig interactief bedieningspaneel waarbij knopwaarden meters aansturen, switches LEDs doden en vier tegelijk maxen een noodalarm triggert — puur via de cascade en selectors — had ik niet voor mogelijk gehouden.

Het concept hielp enorm. Door een duidelijk thema te kiezen — een verouderde kerncentrale — kreeg elk CSS-probleem een narratieve context. De koffievlek is niet zomaar een cirkel; de worn-off Pump B label is geen random easter egg maar iets wat je in een echte verlaten controlekamer zou vinden. Dat maakte elke technische beslissing ook een ontwerpbeslissing.

De 9,5 voelt verdiend, maar ook als aanmoediging: de halve punt die er af ging laat ruimte voor wat ik nog beter kan. De file-restructuring kostte vier uur — meer dan ik had verwacht. Eerder structureren scheelt tijd achteraf.

De opdracht

Het web is voor iedereen. Dat klinkt simpel, maar in de praktijk betekent het dat je website moet werken op oude browsers, nieuwe browsers, telefoons, game consoles, met screenreaders, met een muis, met alleen een toetsenbord — en alles daartussenin.

De eindopdracht: zet een officiële belasting-PDF om naar een interactief webformulier. Aangifte erfbelasting — een van de meest complexe formulieren die burgers moeten invullen. Het uitgangspunt: HTML first, dan CSS, dan JavaScript. Elk niveau moet zelfstandig bruikbaar zijn.

Wat ik heb gebouwd

Een volledig meerstappe formulier voor de aangifte erfbelasting 2025, gebaseerd op het officiële Belastingdienst-formulier. Het formulier begeleidt de gebruiker stap voor stap door 9 secties, met real-time validatie, een voortgangsbalk, print-styling en donkere modus. De stijl is gebaseerd op het officiële NS design system.

NS design tokens Progressive enhancement localStorage draft Print CSS Dark / light mode Elfproef BSN-validatie ValidityState API fieldset / legend aria-live validatie <dialog> foutmeldingen datalist landenkeuze SVG voortgangsbalk

9-staps formulier

Opgedeeld per logisch onderdeel: gegevens overledene, burgerlijke staat, kinderen, testament, gemachtigde en verkrijgers. Elke stap toont alleen wat relevant is voor de situatie van de gebruiker.

Elfproef BSN-validatie

Het BSN wordt gevalideerd met het elfproef-algoritme — hetzelfde algoritme dat de overheid gebruikt. Eerste check: formaat. Tweede check: wiskundige geldigheid via de weging 9-8-7-6-5-4-3-2-1.

Real-time inline validatie

Elk veld geeft direct feedback bij blur en input. Groen vinkje bij correct, rode foutmelding bij fout — met aria-live="polite" zodat screenreaders meeluisteren.

localStorage concept-opslag

Het formulier slaat automatisch alle veldwaarden op als concept. De gebruiker kan het tabblad sluiten en later verder gaan waar hij gebleven was.

Print styling

Via @media print wordt het antwoordenoverzicht omgezet naar een document dat lijkt op een officieel overheidsdocument. Navigatie, knoppen en decoratie verdwijnen.

NS design system

Alle stijl is gebaseerd op de officiële NS design tokens — kleurenschaal, typografie, spacing, border radius, shadows. Het formulier ziet eruit als iets dat NS zelf zou maken.

Voortgangsbalk met SVG

Een visuele voortgangsbalk bovenaan toont welke stap actief is. De verbindingslijnen tussen de stapindicatoren zijn getekend met inline SVG — iets nieuws voor mij.

Conditionele vragen

Vragen over kinderen, testament en testamentaire erfgenamen verschijnen alleen als de vorige vraag dat vereist. De volgende-vraag-knop verschijnt pas als de huidige vraag beantwoord is.

Code highlight: elfproef BSN-validatie

Een BSN is pas geldig als het elfproef-algoritme een som deelbaar door 11 oplevert. De weging loopt van 9 tot 2 voor de eerste 8 cijfers, met -1 voor het 9e. Dat klonk abstract — totdat ik het stap voor stap uitschreef: 

// Elfproef-validatie voor BSN
// Bron: https://nl.wikipedia.org/wiki/Elfproef
const elfValidBsn = (bsn) => {
  if (!/^\d{8,9}$/.test(bsn)) return false;

  // Pad naar 9 cijfers (8-cijferige BSN's beginnen met 0)
  const digits = bsn.padStart(9, "0").split("").map(Number);

  const sum = digits.reduce((acc, digit, index) => {
    const weight = index === 8 ? -1 : 9 - index;
    // Gewichten: 9, 8, 7, 6, 5, 4, 3, 2, -1
    return acc + digit * weight;
  }, 0);

  return sum % 11 === 0;
};

Code highlight: inline validatie met aria-live

Foutmeldingen die pas verschijnen na het versturen zijn frustrerend. Elk veld valideert op blur, maar ook on input zodat feedback direct verschijnt. De feedback-span wordt dynamisch aangemaakt als die nog niet bestaat, en is voorzien van aria-live="polite" voor screenreaders: 

// Feedback span zoeken of aanmaken naast het veld
const getFeedback = (el) => {
  let next = el.nextElementSibling;
  while (next) {
    if (next.classList.contains("inline-feedback")) return next;
    if (["INPUT", "LABEL", "SELECT"].includes(next.tagName)) break;
    next = next.nextElementSibling;
  }
  // Bestaat nog niet — maak hem aan
  const feedback = document.createElement("span");
  feedback.className = "inline-feedback";
  feedback.setAttribute("aria-live", "polite");
  el.insertAdjacentElement("afterend", feedback);
  return feedback;
};

// Validatie van voorletters (regex: elke letter gevolgd door een punt)
byId("voorletters-overledene").addEventListener("blur", function() {
  if (!/^([A-Za-z]\.)+$/.test(this.value.trim())) {
    markInvalid("voorletters-overledene", "Vul in als: A.B.");
  } else {
    markValid("voorletters-overledene");
  }
});

Code highlight: localStorage concept-opslag

Het formulier itereert alle velden en slaat naam + waarde op bij elke wijziging. Bij terugkeer worden de waarden hersteld — inclusief radio buttons: 

// Alle velden serialiseren naar localStorage
const saveFormDraft = () => {
  const draft = {};
  formElement.querySelectorAll("input, select, textarea").forEach((field) => {
    if (!field.name || field.type === "file") return;
    if (field.type === "radio") {
      if (field.checked) draft[field.name] = field.value;
      return;
    }
    draft[field.name] = field.value;
  });
  localStorage.setItem("erfbelasting-form-draft", JSON.stringify(draft));
};

Nieuwe inzichten

01

Formulieren zijn complexer dan je denkt

Ik dacht dit al te kunnen. Maar als je een écht formulier bouwt — met conditionele vragen, validatie op meerdere niveaus en accessibility — merk je hoe veel er mis kan gaan. Dat inzicht is de kern van dit vak.

02

HTML biedt meer dan je gebruikt

fieldset, legend, datalist, dialog, autocomplete, inputmode — allemaal native HTML features die UX verbeteren zonder een regel JavaScript. Progressive enhancement werkt als je HTML serieus neemt.

03

Validatie is een UX-beslissing

Wanneer toon je een foutmelding? Direct bij typen, bij blur, bij submit? Elke keuze heeft gevolgen voor hoe frustrerend of behulpzaam het formulier aanvoelt. Er is geen universeel juist antwoord — het hangt af van het veldtype.

04

localStorage als no-cost feature

Automatisch concept opslaan kost bijna geen code maar levert enorm veel UX op. Niemand wil een lang formulier opnieuw beginnen omdat hij het tabblad sloot. Dit is een standaard patroon dat ik voortaan altijd toepas.

05

Een design system gebruiken leert je patronen

Het NS design system omzetten naar CSS tokens leerde me hoe je afstand en kleur systematisch organiseert. Het duurde langer dan verwacht, maar het resultaat is aanzienlijk consistenter dan eigen willekeurige waarden.

06

SVG voor custom UI elementen

Voor de verbindingslijnen in de voortgangsbalk had ik inline SVG nodig. Dat was nieuw voor mij — ik wist dat SVG bestond maar had het nog nooit bewust gebruikt om iets te tekenen in de UI.

Reflectie

Browser Technologies was het vak waarbij ik het meest verrast werd door mezelf — in positieve zin. Aan het begin dacht ik formulieren al te begrijpen. Na zes weken bouwen aan één complex formulier besef ik hoe naïef dat was.

Het leukste deel was het NS design system toepassen. Niet omdat het snel ging (dat deed het niet), maar omdat ik voor het eerst een professioneel design system van binnenuit leerde kennen. Je begrijpt pas waarom tokens bestaan als je ze gebruikt om 30 velden consistent te stijlen.

De elfproef was een moment waarop techniek en werkelijkheid samenkwamen. Een algoritme van papier naar werkende code brengen — en het dan ook écht zien werken met testgetallen van de Belastingdienst — was bevredigend op een manier die ik niet had verwacht van een veldvalidatie.

De 10 is mooi. Maar de feedback dat ik nu écht begrijp hoe complex formulieren zijn voelt als de eigenlijke beloning.
