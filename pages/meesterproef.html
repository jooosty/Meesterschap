Meesterproef

Mijn eindproject: een roterende wereldbol en datapijplijn voor de Visdeurbel-datavisualisaties, gebouwd in opdracht van Studio MOAN. Hieronder de opdracht, mijn inbreng en een eerlijke reflectie.

Deadline

19 JUN

Duur

5 weken

Team

04 · CMD

Status

AFGEROND

Opdracht

De Visdeurbel in Utrecht laat je op een bel drukken zodra er een vis voor de sluis verschijnt, zodat de sluiswachter open kan doen. Elke beldruk, elke gespotte vis en elk weggeklikt scherm wordt anoniem gelogd, en die logs groeien aan tot honderden megabytes ruwe JSON. Studio MOAN vroeg ons om daar iets van te maken: niet zomaar de cijfers tonen, maar de data voelbaar maken voor iedereen die de Visdeurbel volgt, van jong tot oud, in de bestaande huisstijl.

Met z'n vieren bouwden we één scrollytelling-website waarin elk hoofdstuk één ding vertelt met één visualisatie. Mijn onderdeel is de roterende wereldbol plus de datapijplijn eronder, die laat zien waar de bezoekers vandaan komen en hoe levend de sluis eigenlijk is.

Mijn rol & de stack

In een team van vier (Julius, Mitchell, Nienke en ik) heb ik de geografische laag voor mijn rekening genomen: de bol, de landenlijst, de tooltips, de kaartmodi én de Python-aggregatie die de ruwe data klein genoeg maakt voor de browser.

React D3.js TopoJSON Vite Python (aggregatie) SVG Orthografische projectie requestAnimationFrame Great-circle arcs Toetsenbord & screenreader

De Visdeurbel-site is groepswerk: vier studenten CMD die elk een eigen visualisatie bouwden, samen op één scrollytelling-pagina in de huisstijl van de Visdeurbel. Mijn wereldbol is dus één hoofdstuk in een groter verhaal. Wat de anderen maakten:

De roterende wereldbol met drie leesmodi en de datapijplijn eronder, plus de landenlijst, tooltips en de stromen naar Utrecht.

Opdrachtgever: Studio MOAN, Utrecht (contact via Cyd Stumpel). Begeleiding: Jad Joubran (toegankelijkheid) en Vasilis van Gemert (design).

Wat de bol kan

Roterende 3D-bol

Orthografische D3-projectie die je met muis, touch en scroll draait en inzoomt. Auto-rotatie pauzeert zodra je 'm aanraakt en hervat na vier seconden stilte.

Fly-to per land

Klik een land in de lijst ernaast aan en de bol animeert soepel naar dat land toe, ook via toetsenbord met Enter of spatie.

Drie leesmodi

Bezoeken + Lijnen (choropleth met stromen naar Utrecht), Vissoort en Tijdstip. Eén dispatcher kiest de juiste renderer.

Stromen naar Utrecht

Geanimeerde great-circle-lijnen lopen vanaf elk land naar een pulserend Visdeurbel-logo-baken op Utrecht.

Slanke datapijplijn

Een Python-stap aggregeert ~288 MB ruwe NDJSON tot ~40 KB JSON, zodat de browser alleen de getallen krijgt die de grafieken nodig hebben.

Toegankelijke lijst

De landenlijst is volledig toetsenbord- en schermlezer-bedienbaar: ARIA-labels, logische tabvolgorde en Escape om de lijst te sluiten.

Code highlight: de bol opzetten

De kern is een geoOrthographic-projectie. Die geeft een echte bol in plaats van een platte kaart, met clipAngle(90) om de achterkant weg te snijden: 

// orthografische projectie = een echte bol i.p.v. platte kaart
const proj = d3.geoOrthographic()
  .scale(INIT_R)                  // R * 2.4, vult het paneel
  .translate([W / 2 H / 2])
  .rotate([-5 -52 0])           // Europa front-and-centre bij laden
  .clipAngle(90);                // snij de achterkant van de bol weg

const path = d3.geoPath(proj);

// oceaan, graticule (coördinatenraster) en landen tekenen
oceanG.append('path').datum({ type: 'Sphere' }).attr('d' path);
gridG .append('path').datum(d3.geoGraticule()()).attr('d' path);
landG .selectAll('path').data(features).join('path')
      .attr('d' path).attr('fill' C.land);

Code highlight: slepen & auto-rotatie

Slepen roteert de bol met 0.3° per pixel; hoger voelde chaotisch. De auto-rotatie draait via een requestAnimationFrame-loop en pauzeert zodra je iets doet: 

// muisdrag → roteer de bol, 0.3° per pixel
function onDrag(e) {
  const dx = e.clientX - start[0];
  const dy = e.clientY - start[1];
  const r0 = startRotation;
  projRef.current.rotate([r0[0] + dx * 0.3 r0[1] - dy * 0.3]);
  redrawAll();                    // herproject alle landen + lijnen
}

// auto-rotatie: pauzeert bij interactie, hervat na 4s stilte
function tick(now) {
  if (!dragging && now - lastInput > 4000) {
    const [l, p] = projRef.current.rotate();
    projRef.current.rotate([l + 0.012 * dt, p]);
    redrawAll();
  }
  requestAnimationFrame(tick);
}

Code highlight: lijnen naar Utrecht

Per land bemonster ik de great-circle naar Utrecht en buig die naar buiten met een sinus, zodat de lijn als een boog over de bol loopt. Twaalf punten bleken op bolschaal genoeg, een fractie van het oorspronkelijke renderwerk: 

// bemonster de great-circle van land → Utrecht en buig 'm naar buiten
const interp = d3.geoInterpolate([lon, lat], UTRECHT);
const steps  = 12;               // genoeg op bolschaal, minimaal renderwerk
const points = [];

for (let i = 0; i <= steps; i++) {
  const t  = i / steps;
  const pt = proj(interp(t));     // [lon,lat] → schermpositie
  const lift = maxLift * Math.sin(t * Math.PI);  // 0 aan uiteindes, max in midden
  points.push(liftRadially(pt, lift));
}

Code highlight: drie kaartmodi

In plaats van een grote if/else heeft elke modus zijn eigen render-functie. Een dispatcher-object koppelt de moduskey aan de juiste renderer, zodat GlobeMap alleen maar de juiste functie hoeft aan te roepen: 

// één renderer per modus; GlobeMap kiest simpelweg de juiste
export const MODE_RENDERERS = {
  choropleth_flows: renderChoroplethFlows,  // bezoeken + lijnen naar Utrecht
  fish:             renderFish,             // kleur op meest geziene vissoort
  time:             renderTime,             // kleur op gemiddeld actief uur
};

// in GlobeMap.jsx:
(MODE_RENDERERS[mode] || renderChoroplethFlows)(ctx);

Code highlight: de datapijplijn

Het zwaarste werk gebeurt buiten de browser. aggregate() rolt de ruwe events op tot tellingen per land; de vissoort zit verstopt in de referrer-query. Omdat zwaar rekenwerk in een aparte stap zit, blijft de frontend puur tekenen: 

// 288 MB ruwe NDJSON → compacte tellingen per land (~40 KB)
export function aggregate(events) {
  const map = {};

  events.forEach(ev => {
    const numId = ALPHA2_TO_NUMERIC[ev.country];
    if (!numId) return;                  // sla onbekende landen over

    const c = (map[numId] ??= newCountry(ev));
    c.events++;
    if (ev.event_name === 'uploadedFish') c.uploaded++;

    // vissoort zit verstopt in de referrer-query: ?fish=Baars
    const m = (ev.referrer_query || '').match(/fish=([^&]+)/);
    if (m) {
      const f = decodeURIComponent(m[1]);
      c.fish[f] = (c.fish[f] || 0) + 1;
    }
  });

  return map;
}

Code highlight: D3 + React zonder stale closures

Halverwege migreerde ik mijn vanilla-versie naar React. De grootste valkuil: laat D3 de SVG beheren via refs en React de omliggende UI, en lees in event-handlers altijd ref.current in plaats van een waarde uit de init-closure: 

// D3 beheert de SVG via refs, React de omliggende UI
const projRef     = useRef(null);
const initialized = useRef(false);

useEffect(() => {
  if (initialized.current) return;       // geen dubbele init in StrictMode
  initialized.current = true;
  projRef.current = d3.geoOrthographic()/* ... */;
}, []);

// FOUT: handler sluit over de oude `proj` → stale closure
svg.on('wheel' () => zoom(proj));
// GOED: lees projRef.current ín de handler zelf
svg.on('wheel' () => zoom(projRef.current));

Code highlight: de toegankelijke landenlijst

Elk landkaartje is een echte knop voor de screenreader, met een volledige gesproken label. De tabvolgorde wordt gestuurd met tabIndex en Escape sluit de uitgeklapte lijst en stuurt de focus terug naar de knop: 

<div
  className="country-card"
  role="button"
  tabIndex={expanded ? 0 : -1}      /* uit tabvolgorde als de lijst dicht is */
  aria-label={`Nummer ${i + 1}. Land: ${c.name}. Vissen gespot: ${n(c.uploaded)}.`}
  onClick={() => onFlyTo(c)}
  onKeyDown={e => {
    if (e.key === 'Enter' || e.key === ' ') onFlyTo(c);
    if (e.key === 'Escape') collapse();   /* sluit lijst, focus terug op knop */
  }}
>
  <span aria-hidden="true">{flag(c.code)} {c.name}</span>
</div>

Problemen & oplossingen

Zoom/drag kapot na moduswissel

Event-handlers sloten over de initiële proj-variabele. Alle handlers herschreven zodat ze projRef.current lezen.

Bol sprong terug naar het midden

D3 overschreef het viewBox-attribuut bij elke mount. Opgelost door JSX de enige eigenaar van de viewBox te maken.

Knop 128e in de tabvolgorde

Verborgen kaartjes hadden nog tabIndex=0. Met tabIndex=-1 zolang de lijst dicht is, springt Tab er netjes overheen.

Animatie te traag bij veel lijnen

80 punten per arc × alle landen werd een zware loop. Terug naar 12 punten, glow-filter van de paden gehaald en DOM-nodes gecachet.

Witruimte naast de kaart

De max-width op main kneep de bol af. Structurele fix: de component buiten de container plaatsen i.p.v. CSS-trucs.

"unknown" in de vis-legenda

Stond hardgecodeerd in FISH_COLORS. Verwijderd en overal gefilterd via UNKNOWN_VALS.

Nieuwe inzichten

01

D3 en React kunnen samen

Zolang D3 de SVG volledig beheert via refs en React de omliggende UI. Lees in handlers altijd ref.current anders loop je tegen stale closures aan, veruit de meest voorkomende bug bij D3-in-React.

02

Scheid data van visualisatie

Zwaar rekenwerk hoort in een build-stap (Python), de frontend tekent alleen. Bonus: omdat de code de lengte van de data-arrays uitleest, werkt dezelfde visualisatie automatisch voor week- én maanddata.

03

Toegankelijkheid zit in details

Tabvolgorde volgt de DOM, niet de visuele order. tabIndex=-1 haalt elementen netjes uit de tabvolgorde en :focus-visible toont de focusring alleen bij toetsenbordgebruik.

04

Performance is meten, niet gokken

Minder arcpunten (12 i.p.v. 80) en één blur-filter op een klein punt in plaats van op lange paden scheelden enorm per frame, zonder zichtbaar verschil op bolschaal.

05

Gebruikerstests verslaan intuïtie

Mijn aanname dat de modusknoppen bovenaan logischer waren, klopte niet. Onderin gecentreerd bleek prettiger en stond de data niet in de weg.

06

Schrappen is ook ontwerpen

Van tien kaartmodi naar drie, de platte kaart en OS/browser-data eruit. Wat overbleef werd rustiger en voor veel meer mensen te volgen.

Reflectie

Het lastigste was niet de techniek maar de keuze wát ik wel en niet liet zien. Mijn eerste versie had zo'n tien kaartmodi en voelde als een dashboard, cool, maar je wist niet waar je naar keek. De feedback van Studio MOAN ging steeds over hetzelfde: maak het leesbaar, maak het voor iedereen, hou je aan de huisstijl. Daardoor heb ik flink geschrapt: de platte kaart, de OS- en browsermodi en de helft van de visualisaties zijn eruit gegaan, en wat overbleef werd rustiger en duidelijker.

Technisch was de React-migratie de grootste sprong; stale closures en terugkerende full-bleed-layoutbugs hebben me het meeste tijd gekost, maar leerden me ook het meest. Achteraf ben ik het meest tevreden over de performance-slag op de flow-animatie en over het toegankelijk maken van de landenlijst, precies de learning goals die ik me had voorgenomen rond schaalbaarheid, stabiliteit en inclusief ontwerp.

Learning goals

  • Verdiepen in architectuur en schaalbaarheid: Voor de meesterproef wil ik bewuste architectuurkeuzes maken die schaal en onderhoudbaarheid ondersteunen. Ik ga modules en boundaries expliciet definiëren zodat nieuwe features zonder regressies toegevoegd kunnen worden. Daarbij documenteer ik ontwerpkeuzes zodat later terugkijken makkelijk is.
  • Productieklare performance en stabiliteit: De focus ligt op stabiele data-pipelines en predictieve foutenafhandeling zodat de app betrouwbaar draait onder load. Ik implementeer monitoring en geautomatiseerde tests voor kritieke paden. Daarmee verklein ik de kans op productie-incidenten en maak ik performance reproduceerbaar.
  • Toegankelijkheid en inclusief ontwerp: De meesterproef moet voor iedereen bruikbaar zijn; ik bouw vanaf het begin met semantische HTML en toegankelijke interacties. Ik plan gebruikerstests en screenreader-checks in de ontwikkelfases. Zo verzeker ik dat UX-beslissingen inclusief zijn en meetbaar verbeterd worden.
