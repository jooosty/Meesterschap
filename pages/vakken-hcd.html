Sector 05c — Human Centered Design

Human
Centered Design

Vijf iteraties co-design met Berend — blind, NVDA-gebruiker. Het doel: lange spraakberichten opsplitsen zodat hij per onderwerp kan reageren.

9 / 10

HCD

Gebruiker

BEREND

Iteraties

Het probleem

Berend is blind en communiceert dagelijks via spraakberichten. Het probleem: die berichten bevatten vaak meerdere onderwerpen tegelijk. Hij moet het hele bericht horen, alles onthouden, en dan één monolithisch antwoord teruggeven. Er is geen manier om per onderwerp te reageren.

De vraag: hoe kunnen we het voor Berend mogelijk maken om op elk onderwerp in een spraakbericht afzonderlijk te reageren?

De oplossing: Spraakberichten-splitter

Een prototype dat Berend in staat stelt een lang spraakbericht op te nemen of een transcript in te plakken, het bericht automatisch te splitsen in afzonderlijke zinnen, op elke zin afzonderlijk te reageren via tekst of spraak, en alles tegelijk in te dienen.

Exclusive Design Principles

Study Situation

Vijf directe testsessies met Berend — geen aannames, maar co-design met de persoon die het product elke dag zou gebruiken. NVDA, Tab-navigatie, korte aankondigingen: alles ontdekt door te observeren, niet door te raden.

Ignore Conventions

Geen traditioneel formulier. Tab is primair, Enter activeert handelingen, boodschappen sluiten automatisch, alle antwoorden worden tegelijk verstuurd via Ctrl+Enter. Alles anders dan standaard — bewust.

Prioritise Identity

Spraak is standaard, niet tekst. Geen overbodige visuele elementen. Korte aankondigingen: "Antwoord op bericht 2" — nooit de hele berichttekst herhalen. Alles geoptimaliseerd voor Berend, niet voor een generieke gebruiker.

Add Nonsense

Geluidje bij versturen via Web Audio API. Bericht-nummers in plaats van generieke termen. Automatisch focus management. Chat-stijl antwoordenpagina geïnspireerd op imBee — vertrouwd en herkenbaar.

De vijf iteraties

01

30 – 31 maart 2026

Het begin — splitsen en kopiëren

  • Spraakopname en transcript plakken gebouwd
  • Automatische zinssplitsing
  • Antwoordvelden per zin
  • Kopieerknop (per item)

Berend leerde ons: de kopieerknop was nutteloos — hij wil antwoorden versturen, niet kopiëren. Tab-navigatie werkte niet logisch. Automatische focus helpt enorm.

Berend's feedback

"Je moet me vertellen wat je aan het doen bent. Ik kan het niet zien."

02

6 – 7 april 2026

Tab-navigatie centraal

  • Kopieerknop verwijderd
  • Boodschappen tabbaar gemaakt — Tab gaat door alle boodschappen
  • Enter opent een boodschap, focus gaat naar Spreek-in knop
  • Aria-labels bij tabben: berichttekst wordt voorgelezen
  • Boodschappen sluiten automatisch als je wegtabt zonder iets in te vullen

Berend's feedback

"Tabbing voelt als thuiskomen."

03

20 – 21 april 2026

Bulk-actie en auditieve feedback

  • Versturen-knop die alles tegelijk indient
  • Sneltoets Ctrl+Enter om overal ter plekke te versturen
  • Geluidje via Web Audio API + screenreader-aankondiging bij versturen
  • Alle boodschappen voorlezen bij tabben

Berend's feedback

"Auditieve bevestiging werkt beter dan woorden alleen."

04

Finale tuning

Dubbele aankondigingen voorkomen

  • Timer maakt geen geluid meer — puur visueel
  • Antwoordveld zegt alleen "Antwoord op bericht 2", niet de volledige berichttekst
  • Beantwoorden-knop aria-label: "Beantwoorden op bericht [nummer]" — geen herhaling

Berend's feedback

"Ik zou dit zo gebruiken als het live zou gaan. Het voelt klaar."

05

6 mei 2026

Antwoordenpagina als losse chatpagina

  • Antwoordenoverzicht verplaatst naar volledig aparte HTML-pagina (antwoorden.html)
  • Gegevens doorgegeven via sessionStorage — geen koppeling tussen bestanden op runtime
  • Chat-stijl interface geïnspireerd op imBee: groene achtergrond, witte inkomende bubbles, groene uitgaande bubbles
  • Elke antwoordbubble bevat geciteerd origineel bericht voor context
  • Alle originele berichten bovenaan, alle antwoorden gegroepeerd onderaan
  • Berichten zonder antwoord stilletjes overgeslagen — geen overbodige "geen antwoord" melding

Belangrijkste inzicht

"Minder pagina-elementen op één scherm = minder cognitieve belasting voor de screenreader-gebruiker."

Code highlight: double rAF voor screenreader-aankondigingen

NVDA, JAWS en VoiceOver lezen aria-live regions niet altijd opnieuw als de tekst gelijk wordt vervangen. De oplossing: eerst leegmaken, dan twee animatieframes wachten voordat de nieuwe tekst er in komt: 

// main.js — double rAF zorgt dat NVDA/JAWS/VoiceOver de regio altijd herleest
function announce(message) {
  announcer.textContent = "";
  requestAnimationFrame(() => {
    requestAnimationFrame(() => {
      announcer.textContent = message;
    });
  });
}

Code highlight: geluidje via Web Audio API

Berend wil auditieve bevestiging bij het versturen — geen browser-alert, geen tekst, maar een vriendelijk "ding". Gebouwd zonder externe library, puur via de Web Audio API: 

// main.js — sine wave die van 520Hz naar 780Hz ramt in 80ms
function playSendSound() {
  const ctx  = new (window.AudioContext || window.webkitAudioContext)();
  const osc  = ctx.createOscillator();
  const gain = ctx.createGain();
  osc.connect(gain);
  gain.connect(ctx.destination);
  osc.type = "sine";
  osc.frequency.setValueAtTime(520, ctx.currentTime);
  osc.frequency.exponentialRampToValueAtTime(780, ctx.currentTime + 0.08);
  gain.gain.setValueAtTime(0.18, ctx.currentTime);
  gain.gain.exponentialRampToValueAtTime(0.001, ctx.currentTime + 0.25);
  osc.start(ctx.currentTime);
  osc.stop(ctx.currentTime + 0.25);
}

Code highlight: sessionStorage als pagina-brug

De antwoordenpagina is volledig los van index.html. Gegevens worden via sessionStorage doorgegeven — zo heeft de screenreader op de nieuwe pagina een schone start zonder de ruis van alle vorige elementen: 

// main.js — sla data op vóór navigatie
sessionStorage.setItem(
  "spraakberichten_antwoorden",
  JSON.stringify(allMessages)
);
window.location.href = "antwoorden.html";

// antwoorden.js — lees data op nieuwe pagina
const raw = sessionStorage.getItem("spraakberichten_antwoorden");
if (raw) allMessages = JSON.parse(raw);

Design keuzes en waarom

Keuze Waarom
Tab-navigatie primair Berend navigeert exclusief met toetsenbord — muis bestaat niet in zijn workflow
Enter = Actie Voelt naturlijker dan standaard "activeren" — minder stappen per handeling
Spraak vóór tekst Berend communiceert liever gesproken — de Spreek-in knop is de eerste optie
Korte aria-labels "Antwoord op 2" i.p.v. volledige berichttekst — geen dubbele aankondigingen
Geluidje bij versturen Auditieve bevestiging voelt zekerder dan tekst alleen — ontdekt in iteratie 3
Automatisch sluiten bij wegtabben Minder navigatiestappen, intuïtief — geen "annuleer" knop nodig
Ctrl+Enter sneltoets Voor ervarenen, verborgen voor beginners — Berend begreep het meteen
Aparte antwoordenpagina Screenreader start opnieuw, zonder ruis van vorige elementen
Antwoorden onderaan gegroepeerd Eerst lezen, dan overzien — logische flow voor Berend
Geen "geen antwoord" melding Overbodige ruis weghalen = rustiger screenreader-ervaring

Feedback

Officiële beoordeling · 9 / 10
"Echt heel erg goed. Een enorme attention to detail, die met elke iteratie groeide. Goed proces, en een goede reflectie."

Nieuwe inzichten

01

Co-design met één persoon werkt

Je leert diepere dingen dan bij tien generieke testers. Eén persoon die het product echt nodig heeft vertelt je dingen die geen user-story ooit zou bevatten.

02

Auditieve feedback is niet optioneel

Voor screenreader-gebruikers is auditieve feedback net zo cruciaal als visuele feedback voor ziende gebruikers. Het geluidje bij versturen deed meer dan de tekst "Antwoorden verstuurd" ooit kon.

03

Minder is meer

Verwijder de kopieerknop. Sluit boodschappen automatisch. Zeg geen dubbele dingen. Elke element dat je weglaat is een navigatiestap minder voor iemand die de hele pagina met Tab doorloopt.

04

Tab is de interface

Als je Tab snapt, bouw je voor screenreader-gebruikers die Tab prefereren. Tabvolgorde is geen detail — het is het ontwerp zelf.

05

Paginagrenzen helpen

Een aparte pagina voor het antwoordoverzicht geeft de screenreader een schone start. De mentale grens voor Berend was ook duidelijker: eerst reageren, dan overzien.

06

Conventions bestaan voor iedereen

Sommige webconventies zijn gemaakt door mensen voor mensen die er op dezelfde manier in navigeren. Exclusive Design leerde me te vragen: werkt deze conventie ook voor Berend? Soms niet. Dan herzie je hem.

Reflectie

HCD was het vak dat het meest van mij vroeg op persoonlijk vlak. Niet de code was complex, maar het afstemmen op één specifieke persoon. Het eerste instinct bij feedback van Berend was verdedigen — "maar het werkt toch?". Het tweede instinct was luisteren.

De vijf iteraties laten goed zien hoe het product transformeerde. Wat begon als een functioneel prototype werd iets dat Berend herkende als zijn manier van werken. Het moment waarop hij zei "het voelt klaar" was het beste compliment dat ik bij dit vak kon krijgen.

De feedback over "attention to detail die met elke iteratie groeide" klopt. Je gaat details zien die je normaal nooit zou opmerken — dubbele aria-aankondigingen, de volgorde van Tab-stops, de timing van een geluidje. Dat niveau van aandacht neem ik mee naar elk volgend project.
